    Keeping Your Python Codebase Tidy: Linters, Type Checkers, and Workspace Harmony

    After fifteen posts about architecture, patterns, and deployment, let's talk about something equally important but often overlooked: keeping your codebase consistently clean as it grows. I've seen brilliant architectures deteriorate not from bad design decisions, but from gradual erosion of code quality standards. A thousand small inconsistencies eventually make a codebase unmaintainable.

    The clean-py repository uses multiple complementary tools to maintain code quality. This isn't the only way—it's one approach I've found convenient after years of experimentation. Your team might prefer different tools or workflows, and that's perfectly fine. The key is having a system and sticking to it.

    The Multi-Tool Approach: Why One Tool Isn't Enough

    Each code quality tool serves a different purpose. Trying to find one tool that does everything is like looking for a universal remote that also makes coffee—theoretically possible, but probably not optimal at either task.

    Here's the toolkit approach used in clean-py:

    • Black: Formats code (opinions included, debates excluded)
    • isort: Organizes imports consistently
    • Ruff: Fast linting and basic error catching
    • Pylint: Deep code analysis and convention enforcement
    • Pyright: Strict type checking
    • Mypy: Alternative type checking with different strengths
    • Bandit: Security vulnerability scanning
    • Vulture: Dead code detection
    • Radon: Complexity analysis

    Each tool catches different issues. Ruff might miss a design smell that Pylint catches. Pyright might find type issues that Mypy misses (and vice versa). The overlap isn't redundancy—it's defense in depth.

    Pylint: The Thorough Inspector

    Pylint goes beyond syntax checking to enforce coding standards and catch design issues. While Ruff is blazingly fast for basic checks, Pylint takes its time to perform deeper analysis:

    # .pylintrc
[MASTER]
# Specify a score threshold to be exceeded before program exits with error
fail-under=8.0

# Use multiple processes to speed up Pylint
jobs=4

# Pickle collected data for later comparisons
persistent=yes

# Allow loading of arbitrary C extensions
unsafe-load-any-extension=no

[MESSAGES CONTROL]
# Disable messages that conflict with other tools or are too strict
disable=
    C0111,  # missing-docstring (we use interrogate for this)
    C0103,  # invalid-name (too strict for test functions)
    R0903,  # too-few-public-methods (value objects often have few methods)
    R0801,  # duplicate-code (often false positives in tests)
    W0212,  # protected-access (needed for testing private methods)
    E1101,  # no-member (false positives with SQLAlchemy)
    W0511,  # fixme (we track TODOs differently)

[REPORTS]
# Set the output format
output-format=colorized

# Include message's id in output
include-ids=yes

# Include symbolic ids of messages in output
symbol=yes

# Put messages in separate files
files-output=no

# Template for output
msg-template={path}:{line}: [{msg_id}({symbol}), {obj}] {msg}

[BASIC]
# Naming conventions
argument-naming-style=snake_case
attr-naming-style=snake_case
class-attribute-naming-style=any
class-naming-style=PascalCase
const-naming-style=UPPER_CASE
function-naming-style=snake_case
method-naming-style=snake_case
module-naming-style=snake_case
variable-naming-style=snake_case

# Good names which should always be accepted
good-names=i,j,k,ex,Run,_,id,ok,db

# Bad names which should always be refused
bad-names=foo,bar,baz,toto,tutu,tata

[DESIGN]
# Maximum number of arguments for functions
max-args=7

# Maximum number of attributes for a class
max-attributes=10

# Maximum number of boolean expressions in an if statement
max-bool-expr=5

# Maximum number of branch for function/method body
max-branches=12

# Maximum number of locals for function/method body
max-locals=15

# Maximum number of parents for a class
max-parents=7

# Maximum number of public methods for a class
max-public-methods=20

# Maximum number of return/yield for function/method body
max-returns=6

# Maximum number of statements in function/method body
max-statements=50

# Minimum number of public methods for a class
min-public-methods=0

[SIMILARITIES]
# Minimum lines number of a similarity
min-similarity-lines=4

# Ignore comments when computing similarities
ignore-comments=yes

# Ignore docstrings when computing similarities
ignore-docstrings=yes

# Ignore imports when computing similarities
ignore-imports=yes

[TYPECHECK]
# List of class names for which member attributes should not be checked
ignored-classes=optparse.Values,thread._local,_thread._local,SQLAlchemy,scoped_session

# List of module names for which member attributes should not be checked
ignored-modules=alembic.context,sqlalchemy.orm

[IMPORTS]
# Allow wildcard imports from modules
allow-wildcard-with-all=no

# Analyse import fallback blocks
analyse-fallback-blocks=no

# Deprecated modules which should not be used
deprecated-modules=optparse,tkinter.tix

[EXCEPTIONS]
# Exceptions that will emit a warning when being caught
overgeneral-exceptions=BaseException,Exception

    # Pylint catches these design issues

class OrderService:
    def process_order(self, order_id, customer_id, product_ids, 
                      quantities, prices, discounts, shipping_address,
                      billing_address, payment_method):  # too-many-arguments
        """Process an order with too many parameters."""
        pass

def calculate_total(items):
    # too-many-branches, too-many-statements
    total = 0
    for item in items:
        if item.type == 'product':
            if item.taxable:
                if item.category == 'food':
                    total += item.price * 1.05
                elif item.category == 'clothing':
                    total += item.price * 1.08
                else:
                    total += item.price * 1.10
            else:
                total += item.price
        elif item.type == 'service':
            if item.duration > 4:
                total += item.price * 1.5
            else:
                total += item.price
    # ... 20 more conditions
    return total

# Pylint suggests refactoring to:
class PriceCalculator:
    def calculate_total(self, items):
        return sum(self._calculate_item_price(item) for item in items)

    def _calculate_item_price(self, item):
        if item.type == 'product':
            return self._calculate_product_price(item)
        elif item.type == 'service':
            return self._calculate_service_price(item)
        return item.price

    Pyright's advantages include: - Faster performance on large codebases - Better IDE integration (especially VS Code) - More accurate type inference in some cases - Better support for type narrowing

    Example where Pyright shines:

    from typing import Union, Literal

def process_value(value: Union[int, str]) -> str:
    # Pyright understands type narrowing better
    if isinstance(value, int):
        # Pyright knows value is int here
        return str(value + 10)
    else:
        # Pyright knows value is str here
        return value.upper()

# Pyright handles Literal types well
Status = Literal["pending", "approved", "rejected"]

def handle_status(status: Status) -> None:
    # Pyright ensures exhaustive handling
    match status:
        case "pending":
            print("Waiting")
        case "approved":
            print("Good to go")
        # Pyright warns: Case "rejected" not handled

    Make Commands: One Interface to Rule Them All

    The Makefile provides a consistent interface regardless of the underlying tools:

    # Additional Make commands for code maintenance

.PHONY: lint-all
lint-all: ## Run all linting tools
    @echo "Running comprehensive linting..."
    @echo "1/5 Ruff..."
    @ruff check src tests
    @echo "2/5 Pylint..."
    @pylint src --rcfile=.pylintrc
    @echo "3/5 Mypy..."
    @mypy src --strict
    @echo "4/5 Pyright..."
    @pyright src
    @echo "5/5 Bandit..."
    @bandit -r src -ll
    @echo "✅ All linting passed!"

.PHONY: fix-all
fix-all: ## Auto-fix all possible issues
    @echo "Auto-fixing code issues..."
    @black src tests
    @isort src tests
    @ruff check --fix src tests
    @autoflake --in-place --remove-all-unused-imports --recursive src tests
    @echo "✅ Auto-fixes applied!"

.PHONY: complexity
complexity: ## Analyze code complexity
    @echo "Analyzing code complexity..."
    @radon cc src --min B --show-complexity --total-average
    @radon mi src --min B --show
    @echo ""
    @echo "Cyclomatic Complexity:"
    @echo "  A: 1-5   (low risk - simple block)"
    @echo "  B: 6-10  (low risk - well structured)"
    @echo "  C: 11-20 (moderate risk - more complex)"
    @echo "  D: 21-50 (high risk - complex, alarming)"
    @echo "  E: >50   (very high risk - untestable)"

.PHONY: dead-code
dead-code: ## Find dead code
    @echo "Searching for dead code..."
    @vulture src --min-confidence 80

.PHONY: todos
todos: ## List all TODOs and FIXMEs
    @echo "TODOs and FIXMEs:"
    @grep -r "TODO\|FIXME\|HACK\|XXX" src tests --exclude-dir=__pycache__ || echo "No TODOs found!"

.PHONY: deps-check
deps-check: ## Check dependency health
    @echo "Checking dependencies..."
    @pip list --outdated
    @echo ""
    @echo "Security check:"
    @safety check || true
    @echo ""
    @echo "License check:"
    @pip-licenses --with-urls --format=markdown

.PHONY: stats
stats: ## Show codebase statistics
    @echo "Codebase Statistics:"
    @echo "==================="
    @echo "Lines of Code:"
    @cloc src --exclude-dir=__pycache__
    @echo ""
    @echo "File Count:"
    @find src -name "*.py" | wc -l
    @echo ""
    @echo "Test Coverage:"
    @coverage report --skip-covered --sort=cover

.PHONY: health
health: ## Full codebase health check
    @echo "🏥 Running full health check..."
    @$(MAKE) lint-all
    @$(MAKE) test
    @$(MAKE) complexity
    @$(MAKE) dead-code
    @$(MAKE) deps-check
    @echo "✅ Codebase is healthy!"

.PHONY: pre-commit-all
pre-commit-all: ## Run pre-commit on all files
    @pre-commit run --all-files

.PHONY: clean-deep
clean-deep: clean ## Deep clean including caches and environments
    @echo "Deep cleaning..."
    @rm -rf .venv venv
    @rm -rf node_modules
    @rm -rf .tox
    @rm -rf .eggs *.egg-info
    @find . -type d -name ".pytest_cache" -exec rm -rf {} + 2>/dev/null || true
    @find . -type d -name ".mypy_cache" -exec rm -rf {} + 2>/dev/null || true
    @find . -type d -name ".ruff_cache" -exec rm -rf {} + 2>/dev/null || true
    @find . -type d -name "__pycache__" -exec rm -rf {} + 2>/dev/null || true
    @echo "✅ Deep clean complete!"

    Alternative Approaches

    This multi-tool approach works well for the clean-py project, but it's not the only way. Here are alternatives worth considering:

    The Minimalist Approach

    Some teams prefer fewer tools: - Ruff only: Fast and covers most linting needs - Black + Ruff: Formatting plus linting - Poetry + Black + Pytest: Simplified toolchain

    The All-in-One Approach

    Tools that try to do everything: - Prospector: Wraps multiple tools - Sourcery: AI-powered code review - SonarQube: Enterprise code quality platform

    The Progressive Approach

    Start simple, add tools as needed: 1. Start with Black (formatting) 2. Add Ruff (fast linting) 3. Add Mypy (type checking) 4. Add Pylint (deep analysis) when needed 5. Add specialized tools (security, complexity) for mature projects

    Continuous Code Health Monitoring

    Beyond running tools, monitor code health trends:

    # scripts/code_health.py
import json
import subprocess
from datetime import datetime
from pathlib import Path

def collect_metrics():
    """Collect code health metrics."""
    metrics = {
        "timestamp": datetime.now().isoformat(),
        "metrics": {}
    }

    # Complexity metrics
    result = subprocess.run(
        ["radon", "cc", "src", "--json"],
        capture_output=True,
        text=True
    )
    complexity_data = json.loads(result.stdout)

    total_complexity = 0
    high_complexity_functions = 0

    for file_data in complexity_data.values():
        for func in file_data:
            if isinstance(func, dict):
                total_complexity += func.get("complexity", 0)
                if func.get("complexity", 0) > 10:
                    high_complexity_functions += 1

    metrics["metrics"]["average_complexity"] = total_complexity / max(len(complexity_data), 1)
    metrics["metrics"]["high_complexity_functions"] = high_complexity_functions

    # Pylint score
    result = subprocess.run(
        ["pylint", "src", "--output-format=json"],
        capture_output=True,
        text=True
    )
    if result.stdout:
        pylint_data = json.loads(result.stdout)
        metrics["metrics"]["pylint_score"] = pylint_data.get("score", 0)

    # Type coverage (pyright)
    result = subprocess.run(
        ["pyright", "--outputjson"],
        capture_output=True,
        text=True
    )
    if result.stdout:
        pyright_data = json.loads(result.stdout)
        metrics["metrics"]["type_errors"] = pyright_data["summary"]["errorCount"]
        metrics["metrics"]["type_warnings"] = pyright_data["summary"]["warningCount"]

    return metrics

def generate_report(metrics):
    """Generate health report."""
    print("📊 Code Health Report")
    print("=" * 50)
    print(f"Timestamp: {metrics['timestamp']}")
    print(f"Average Complexity: {metrics['metrics'].get('average_complexity', 0):.2f}")
    print(f"High Complexity Functions: {metrics['metrics'].get('high_complexity_functions', 0)}")
    print(f"Pylint Score: {metrics['metrics'].get('pylint_score', 0):.2f}/10")
    print(f"Type Errors: {metrics['metrics'].get('type_errors', 0)}")
    print(f"Type Warnings: {metrics['metrics'].get('type_warnings', 0)}")

    # Save to file for tracking
    history_file = Path("code_health_history.json")
    history = []
    if history_file.exists():
        history = json.loads(history_file.read_text())
    history.append(metrics)
    history_file.write_text(json.dumps(history, indent=2))

if __name__ == "__main__":
    metrics = collect_metrics()
    generate_report(metrics)

    Common Pitfalls and Solutions

    Tool Overload

    Problem: Too many tools slow down development. Solution: Start with essentials, add tools when problems arise.

    Configuration Drift

    Problem: Local configs diverge from team standards. Solution: Version control all configuration files, use workspace settings.

    Ignored Warnings

    Problem: Teams ignore tool output due to noise. Solution: Configure tools properly, fix warnings immediately or suppress with reason.

    Performance Issues

    Problem: Linting takes too long. Solution: Use faster tools (Ruff), run expensive checks only in CI.

    The Human Element

    Tools don't maintain code quality—people do. The best tooling setup fails if the team doesn't buy in. Here's what works:

    1. Explain the Why: Show how tools save time and prevent bugs
    2. Start Gradually: Don't enable all checks at once
    3. Fix Together: Make fixing warnings a team activity
    4. Celebrate Improvements: Share metrics showing quality improvements
    5. Allow Flexibility: Some rules might not fit your project

    Conclusion

    Keeping a Python codebase tidy isn't about following a prescriptive set of rules—it's about finding what works for your team and automating it. The clean-py repository demonstrates one approach using multiple complementary tools, comprehensive configuration, and convenient Make commands.

    The specific tools matter less than having a system. Whether you use Pylint or only Ruff, Pyright or Mypy, VS Code or PyCharm—what matters is consistency, automation, and team buy-in. Start with the basics, add tools as needed, and always prioritize developer experience alongside code quality.

    Remember: Clean code isn't an end goal—it's a means to deliver value efficiently. The best codebase is one that's clean enough to maintain, flexible enough to evolve, and simple enough for your team to understand.

    The tools and configurations shown here are one way of achieving this. Your way might be different, and that's perfectly fine. The important thing is to have a way, document it, automate it, and stick to it.
