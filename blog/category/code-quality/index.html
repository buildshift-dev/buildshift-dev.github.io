    Keeping Your Python Codebase Tidy: Linters, Type Checkers, and Workspace Harmony

    After fifteen posts about architecture, patterns, and deployment, let's talk about something equally important but often overlooked: keeping your codebase consistently clean as it grows. I've seen brilliant architectures deteriorate not from bad design decisions, but from gradual erosion of code quality standards. A thousand small inconsistencies eventually make a codebase unmaintainable.

    The clean-py repository uses multiple complementary tools to maintain code quality. This isn't the only way—it's one approach I've found convenient after years of experimentation. Your team might prefer different tools or workflows, and that's perfectly fine. The key is having a system and sticking to it.

    The Multi-Tool Approach: Why One Tool Isn't Enough

    Each code quality tool serves a different purpose. Trying to find one tool that does everything is like looking for a universal remote that also makes coffee—theoretically possible, but probably not optimal at either task.

    Here's the toolkit approach used in clean-py:

    • Black: Formats code (opinions included, debates excluded)
    • isort: Organizes imports consistently
    • Ruff: Fast linting and basic error catching
    • Pylint: Deep code analysis and convention enforcement
    • Pyright: Strict type checking
    • Mypy: Alternative type checking with different strengths
    • Bandit: Security vulnerability scanning
    • Vulture: Dead code detection
    • Radon: Complexity analysis

    Each tool catches different issues. Ruff might miss a design smell that Pylint catches. Pyright might find type issues that Mypy misses (and vice versa). The overlap isn't redundancy—it's defense in depth.

    Pylint: The Thorough Inspector

    Pylint goes beyond syntax checking to enforce coding standards and catch design issues. While Ruff is blazingly fast for basic checks, Pylint takes its time to perform deeper analysis:

    # .pylintrc
[MASTER]
# Specify a score threshold to be exceeded before program exits with error
fail-under=8.0

# Use multiple processes to speed up Pylint
jobs=4

# Pickle collected data for later comparisons
persistent=yes

# Allow loading of arbitrary C extensions
unsafe-load-any-extension=no

[MESSAGES CONTROL]
# Disable messages that conflict with other tools or are too strict
disable=
    C0111,  # missing-docstring (we use interrogate for this)
    C0103,  # invalid-name (too strict for test functions)
    R0903,  # too-few-public-methods (value objects often have few methods)
    R0801,  # duplicate-code (often false positives in tests)
    W0212,  # protected-access (needed for testing private methods)
    E1101,  # no-member (false positives with SQLAlchemy)
    W0511,  # fixme (we track TODOs differently)

[REPORTS]
# Set the output format
output-format=colorized

# Include message's id in output
include-ids=yes

# Include symbolic ids of messages in output
symbol=yes

# Put messages in separate files
files-output=no

# Template for output
msg-template={path}:{line}: [{msg_id}({symbol}), {obj}] {msg}

[BASIC]
# Naming conventions
argument-naming-style=snake_case
attr-naming-style=snake_case
class-attribute-naming-style=any
class-naming-style=PascalCase
const-naming-style=UPPER_CASE
function-naming-style=snake_case
method-naming-style=snake_case
module-naming-style=snake_case
variable-naming-style=snake_case

# Good names which should always be accepted
good-names=i,j,k,ex,Run,_,id,ok,db

# Bad names which should always be refused
bad-names=foo,bar,baz,toto,tutu,tata

[DESIGN]
# Maximum number of arguments for functions
max-args=7

# Maximum number of attributes for a class
max-attributes=10

# Maximum number of boolean expressions in an if statement
max-bool-expr=5

# Maximum number of branch for function/method body
max-branches=12

# Maximum number of locals for function/method body
max-locals=15

# Maximum number of parents for a class
max-parents=7

# Maximum number of public methods for a class
max-public-methods=20

# Maximum number of return/yield for function/method body
max-returns=6

# Maximum number of statements in function/method body
max-statements=50

# Minimum number of public methods for a class
min-public-methods=0

[SIMILARITIES]
# Minimum lines number of a similarity
min-similarity-lines=4

# Ignore comments when computing similarities
ignore-comments=yes

# Ignore docstrings when computing similarities
ignore-docstrings=yes

# Ignore imports when computing similarities
ignore-imports=yes

[TYPECHECK]
# List of class names for which member attributes should not be checked
ignored-classes=optparse.Values,thread._local,_thread._local,SQLAlchemy,scoped_session

# List of module names for which member attributes should not be checked
ignored-modules=alembic.context,sqlalchemy.orm

[IMPORTS]
# Allow wildcard imports from modules
allow-wildcard-with-all=no

# Analyse import fallback blocks
analyse-fallback-blocks=no

# Deprecated modules which should not be used
deprecated-modules=optparse,tkinter.tix

[EXCEPTIONS]
# Exceptions that will emit a warning when being caught
overgeneral-exceptions=BaseException,Exception

# Pylint suggests refactoring to:
class PriceCalculator:
    def calculate_total(self, items):
        return sum(self._calculate_item_price(item) for item in items)

    def _calculate_item_price(self, item):

    Pyright's advantages include: - Faster performance on large codebases - Better IDE integration (especially VS Code) - More accurate type inference in some cases - Better support for type narrowing

    Example where Pyright shines:

    from typing import Union, Literal

def process_value(value: Union[int, str]) -> str:
    # Pyright understands type narrowing better
    if isinstance(value, int):
        # Pyright knows value is int here
        return str(value + 10)
    else:
        # Pyright knows value is str here
        return value.upper()

# Pyright handles Literal types well
Status = Literal["pending", "approved", "rejected"]

def handle_status(status: Status) -> None:
    # Pyright ensures exhaustive handling
    match status:
        case "pending":
            print("Waiting")
        case "approved":
            print("Good to go")
        # Pyright warns: Case "rejected" not handled

    Alternative Approaches

    This multi-tool approach works well for the clean-py project, but it's not the only way. Here are alternatives worth considering:

    The Minimalist Approach

    Some teams prefer fewer tools: - Ruff only: Fast and covers most linting needs - Black + Ruff: Formatting plus linting - Poetry + Black + Pytest: Simplified toolchain

    The All-in-One Approach

    Tools that try to do everything: - Prospector: Wraps multiple tools - Sourcery: AI-powered code review - SonarQube: Enterprise code quality platform

    The Progressive Approach

    Start simple, add tools as needed: 1. Start with Black (formatting) 2. Add Ruff (fast linting) 3. Add Mypy (type checking) 4. Add Pylint (deep analysis) when needed 5. Add specialized tools (security, complexity) for mature projects

    Continuous Code Health Monitoring

    Beyond running tools, monitor code health trends:

    # scripts/code_health.py
import json
import subprocess
from datetime import datetime
from pathlib import Path

def collect_metrics():
    """Collect code health metrics."""
    metrics = {
        "timestamp": datetime.now().isoformat(),
        "metrics": {}
    }

    # Complexity metrics
    result = subprocess.run(
        ["radon", "cc", "src", "--json"],
        capture_output=True,
        text=True
    )
    complexity_data = json.loads(result.stdout)

    total_complexity = 0
    high_complexity_functions = 0

    for file_data in complexity_data.values():
        for func in file_data:
            if isinstance(func, dict):
                total_complexity += func.get("complexity", 0)
                if func.get("complexity", 0) > 10:
                    high_complexity_functions += 1

    metrics["metrics"]["average_complexity"] = total_complexity / max(len(complexity_data), 1)
    metrics["metrics"]["high_complexity_functions"] = high_complexity_functions

    # Pylint score
    result = subprocess.run(
        ["pylint", "src", "--output-format=json"],
        capture_output=True,
        text=True
    )
    if result.stdout:
        pylint_data = json.loads(result.stdout)
        metrics["metrics"]["pylint_score"] = pylint_data.get("score", 0)

    # Type coverage (pyright)
    result = subprocess.run(
        ["pyright", "--outputjson"],
        capture_output=True,
        text=True
    )
    if result.stdout:
        pyright_data = json.loads(result.stdout)
        metrics["metrics"]["type_errors"] = pyright_data["summary"]["errorCount"]
        metrics["metrics"]["type_warnings"] = pyright_data["summary"]["warningCount"]

    return metrics

def generate_report(metrics):
    """Generate health report."""
    print("📊 Code Health Report")
    print("=" * 50)
    print(f"Timestamp: {metrics['timestamp']}")
    print(f"Average Complexity: {metrics['metrics'].get('average_complexity', 0):.2f}")
    print(f"High Complexity Functions: {metrics['metrics'].get('high_complexity_functions', 0)}")
    print(f"Pylint Score: {metrics['metrics'].get('pylint_score', 0):.2f}/10")
    print(f"Type Errors: {metrics['metrics'].get('type_errors', 0)}")
    print(f"Type Warnings: {metrics['metrics'].get('type_warnings', 0)}")

    # Save to file for tracking
    history_file = Path("code_health_history.json")
    history = []
    if history_file.exists():
        history = json.loads(history_file.read_text())
    history.append(metrics)
    history_file.write_text(json.dumps(history, indent=2))

if __name__ == "__main__":
    metrics = collect_metrics()
    generate_report(metrics)

    Common Pitfalls and Solutions

    Tool Overload

    Problem: Too many tools slow down development. Solution: Start with essentials, add tools when problems arise.

    Configuration Drift

    Problem: Local configs diverge from team standards. Solution: Version control all configuration files, use workspace settings.

    Ignored Warnings

    Problem: Teams ignore tool output due to noise. Solution: Configure tools properly, fix warnings immediately or suppress with reason.

    Performance Issues

    Problem: Linting takes too long. Solution: Use faster tools (Ruff), run expensive checks only in CI.

    The Human Element

    Tools don't maintain code quality—people do. The best tooling setup fails if the team doesn't buy in. Here's what works:

    1. Explain the Why: Show how tools save time and prevent bugs
    2. Start Gradually: Don't enable all checks at once
    3. Fix Together: Make fixing warnings a team activity
    4. Celebrate Improvements: Share metrics showing quality improvements
    5. Allow Flexibility: Some rules might not fit your project

    Conclusion

    Keeping a Python codebase tidy isn't about following a prescriptive set of rules—it's about finding what works for your team and automating it. The clean-py repository demonstrates one approach using multiple complementary tools, comprehensive configuration, and convenient Make commands.

    The specific tools matter less than having a system. Whether you use Pylint or only Ruff, Pyright or Mypy, VS Code or PyCharm—what matters is consistency, automation, and team buy-in. Start with the basics, add tools as needed, and always prioritize developer experience alongside code quality.

    Remember: Clean code isn't an end goal—it's a means to deliver value efficiently. The best codebase is one that's clean enough to maintain, flexible enough to evolve, and simple enough for your team to understand.

    The tools and configurations shown here are one way of achieving this. Your way might be different, and that's perfectly fine. The important thing is to have a way, document it, automate it, and stick to it.

    Code Quality Automation: Pre-commit Hooks and CI/CD

    Code quality isn't about perfection—it's about consistency. I've worked on teams where every developer had their own formatting style, their own import organization, their own naming conventions. Code reviews devolved into style debates instead of logic discussions. Time was wasted on manual formatting instead of solving problems.

    The clean-py repository demonstrates how automation eliminates these issues entirely. Quality checks run automatically. Formatting happens without thinking. Security scans catch vulnerabilities before they reach production. This isn't about replacing human judgment—it's about freeing humans to focus on what matters.

    The Cost of Manual Quality Control

    Let me paint a familiar picture. A developer submits a pull request. The reviewer spots inconsistent formatting. "Please run Black," they comment. The developer formats the code, pushes again. Now there's a linting error. Another round trip. Then someone notices the imports aren't sorted. Another push. Finally, the actual logic review begins—but everyone's already frustrated.

    I've seen this pattern waste days per sprint. Worse, when developers are tired or rushed, quality checks get skipped. "We'll fix it later," becomes technical debt that never gets paid.

    Manual quality control has hidden costs: - Context switching: Developers break flow to run quality tools - Inconsistent application: Some files get checked, others don't - Review fatigue: Reviewers waste energy on mechanical issues - Delayed feedback: Problems found late in the process are expensive to fix - Knowledge silos: Only certain developers know all the quality rules

    Pre-commit Hooks: Your First Line of Defense

    Pre-commit hooks solve these problems by running quality checks automatically before code enters the repository. The clean-py repository uses a comprehensive pre-commit configuration:

    # .pre-commit-config.yaml
default_language_version:
  python: python3.11

repos:
  # General file fixes
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
        args: ['--unsafe']  # Allow custom tags
      - id: check-json
      - id: check-toml
      - id: check-added-large-files
        args: ['--maxkb=1000']
      - id: check-case-conflict
      - id: check-merge-conflict
      - id: detect-private-key
      - id: debug-statements
      - id: mixed-line-ending
        args: ['--fix=lf']

  # Python code formatting
  - repo: https://github.com/psf/black
    rev: 23.11.0
    hooks:
      - id: black
        language_version: python3.11
        args: ['--line-length=88']

  # Python import sorting
  - repo: https://github.com/pycqa/isort
    rev: 5.12.0
    hooks:
      - id: isort
        args: ["--profile", "black", "--line-length", "88"]

  # Python linting with Ruff (fast!)
  - repo: https://github.com/charliermarsh/ruff-pre-commit
    rev: v0.1.6
    hooks:
      - id: ruff
        args: [--fix, --exit-non-zero-on-fix]

  # Security scanning
  - repo: https://github.com/PyCQA/bandit
    rev: 1.7.5
    hooks:
      - id: bandit
        args: ['-r', 'src', '-ll', '--skip', 'B101']

    Python Type Hints: Your First Line of Defense

    Python without type hints is like driving without headlights—you can do it, but you're bound to hit something eventually. I've spent too many hours debugging runtime errors that could have been caught at development time: AttributeError: 'NoneType' object has no attribute 'value', TypeError: unsupported operand type(s) for +: 'str' and 'int', and the classic KeyError: 'customer_id' in production.

    Type hints transform Python development from a guessing game into a predictable, maintainable experience. They serve as documentation, catch bugs early, and make your code self-explaining. The clean-py repository demonstrates how comprehensive type annotations create robust, professional Python applications.

    The Case for Type Hints

    Dynamic typing is Python's superpower and its kryptonite. The flexibility that makes Python productive also makes it fragile. Consider this seemingly innocent function:

    # No type hints - runtime disasters waiting to happen
def calculate_order_total(items, tax_rate, discount):
    subtotal = sum(item.price * item.quantity for item in items)
    discounted = subtotal - discount
    tax = discounted * tax_rate
    return discounted + tax

# What happens when someone calls it like this?
total = calculate_order_total("not a list", "not a number", None)
# Runtime explosion guaranteed

    The problems are invisible until runtime: - What type should items be? - Is discount a percentage or absolute amount? - Can any of these parameters be None? - What does the function return?

    With type hints, these questions have clear answers:

    # Type hints make expectations explicit
from typing import List, Optional
from decimal import Decimal

def calculate_order_total(
    items: List[OrderItem], 
    tax_rate: Decimal, 
    discount: Optional[Decimal] = None
) -> Decimal:
    """Calculate order total with tax and optional discount."""
    subtotal = sum(item.price * item.quantity for item in items)

    if discount is not None:
        subtotal -= discount

    tax = subtotal * tax_rate
    return subtotal + tax

# Now it's impossible to misuse
# calculate_order_total("invalid", "invalid", "invalid")  # Type checker catches this

from dataclasses import dataclass
from decimal import Decimal
from typing import Self  # Python 3.11+

@dataclass(frozen=True)
class Money(ValueObject):
    """Money value object with currency and amount."""

    amount: Decimal
    currency: str

    def add(self, other: "Money") -> "Money":
        """Add two money amounts (must have same currency)."""
        if self.currency != other.currency:
            raise ValueError(
                f"Cannot add different currencies: {self.currency} and {other.currency}"
            )
        return Money(amount=self.amount + other.amount, currency=self.currency)

    def multiply(self, factor: int | Decimal) -> "Money":
        """Multiply money by a factor."""
        if factor < 0:
            raise ValueError("Factor cannot be negative")
        return Money(
            amount=self.amount * Decimal(str(factor)), 
            currency=self.currency
        )

    Protocol Classes for Duck Typing

    Sometimes you need the flexibility of duck typing with the safety of type hints:

@runtime_checkable
class PaymentProcessor(Protocol):
    """Protocol for payment processing."""

    def charge(self, amount: Money, payment_method: str) -> PaymentResult:
        """Process a payment charge."""
        ...

    def refund(self, transaction_id: str, amount: Money) -> RefundResult:
        """Process a refund."""
        ...

class StripeProcessor:
    """Stripe implementation of payment processing."""

    def charge(self, amount: Money, payment_method: str) -> PaymentResult:
        # Stripe-specific implementation
        return PaymentResult(success=True, transaction_id="stripe_123")

    def refund(self, transaction_id: str, amount: Money) -> RefundResult:
        # Stripe-specific refund
        return RefundResult(success=True, refund_id="refund_456")

class PayPalProcessor:
    """PayPal implementation of payment processing."""

        # PayPal-specific implementation

        # PayPal-specific refund
        return RefundResult(success=True, refund_id="pp_refund_101")

# Both implementations satisfy the protocol
def process_payment(
    processor: PaymentProcessor,  # Any object with charge() and refund() methods
    amount: Money,
    payment_method: str
) -> PaymentResult:

    from typing import Callable, Awaitable

class CustomerValidator:
    """Customer validation with typed rules."""

    def __init__(self, rules: List[ValidationRule]) -> None:
        self._rules = rules

        """Validate customer against all rules."""
        return all(rule(customer) for rule in self._rules)

class AsyncCustomerValidator:
    """Async customer validation with typed rules."""

    def __init__(self, rules: List[AsyncValidationRule]) -> None:
        self._rules = rules

    async def validate(self, customer: Customer) -> bool:
        """Validate customer against all async rules."""
        results = await asyncio.gather(*[rule(customer) for rule in self._rules])
        return all(results)

    Gradual Typing Strategy

    Don't try to add type hints everywhere at once. Use a gradual approach:

    Phase 1: Core Domain Models

    Start with your most important business logic:

    # Phase 1: Domain entities and value objects
@dataclass

    customer_id: CustomerId  # Strong typing for identifiers
    name: str
    email: Email  # Value object with validation
    is_active: bool = True

    Phase 2: Repository Interfaces

    Add types to your abstraction boundaries:

    # Phase 2: Repository and service interfaces
class CustomerRepository(ABC):
    @abstractmethod
    async def save(self, customer: Customer) -> Customer:
        pass

    @abstractmethod 
    async def find_by_id(self, customer_id: CustomerId) -> Customer | None:
        pass

    Phase 3: Use Cases and Application Layer

    Type your application workflows:

    # Phase 3: Use cases and application services
class CreateCustomerUseCase:
    def __init__(self, customer_repo: CustomerRepository) -> None:
        self._customer_repo = customer_repo

    async def execute(self, command: CreateCustomerCommand) -> Customer:
        # Typed implementation
        pass

    Phase 4: Infrastructure and Presentation

    Finally, type your outer layers:

    # Phase 4: FastAPI endpoints with full typing

    request: CreateCustomerRequest,

    Common Type Hint Mistakes

    1. Overusing Any

    Any defeats the purpose of type hints:

    # Bad - Any provides no type safety
def process_data(data: Any) -> Any:
    return data.some_method()

# Good - Use specific types or generics
from typing import TypeVar
T = TypeVar('T')

def process_data(data: ProcessableData) -> ProcessedResult:
    return data.process()

# Or use Protocol for duck typing
class Processable(Protocol):
    def process(self) -> ProcessedResult: ...

def process_data(data: Processable) -> ProcessedResult:
    return data.process()

    2. Ignoring None Values

    The most common runtime error in Python:

    # Bad - ignoring potential None
def get_customer_email(customer_id: str) -> str:
    customer = find_customer(customer_id)  # Returns Customer | None
    return customer.email  # ❌ AttributeError if customer is None

# Good - handling None explicitly
def get_customer_email(customer_id: str) -> str | None:
    customer = find_customer(customer_id)  # Returns Customer | None
    if customer is None:
        return None
    return str(customer.email)

# Even better - use Optional handling patterns
def get_customer_email_safe(customer_id: str) -> str:
    customer = find_customer(customer_id)
    if customer is None:
        raise ValueError(f"Customer {customer_id} not found")
    return str(customer.email)

    3. Mutable Default Arguments

    A classic Python gotcha made visible by type hints:

    # Bad - mutable default argument
def create_customer(
    name: str, 
    preferences: dict[str, Any] = {}  # ❌ Shared mutable default
) -> Customer:
    # All customers share the same preferences dict!
    return Customer(name=name, preferences=preferences)

# Good - use None with factory
def create_customer(
    name: str,
    preferences: dict[str, Any] | None = None
) -> Customer:
    if preferences is None:
        preferences = {}  # New dict for each call
    return Customer(name=name, preferences=preferences)

# Best - use factory pattern from dataclass
from dataclasses import field

@dataclass
class Customer:
    name: str
    preferences: dict[str, Any] = field(default_factory=dict)  # ✅ New dict per instance

    Type Hints and Performance

    Type hints have minimal runtime overhead but provide development-time benefits:

    import timeit
from typing import List, Optional

# Function without type hints
def process_items_untyped(items, threshold):
    return [item for item in items if item > threshold]

# Function with type hints
def process_items_typed(items: List[int], threshold: int) -> List[int]:
    return [item for item in items if item > threshold]

# Performance is identical at runtime
items = list(range(10000))
threshold = 5000

time_untyped = timeit.timeit(
    lambda: process_items_untyped(items, threshold), 
    number=1000
)

time_typed = timeit.timeit(
    lambda: process_items_typed(items, threshold),
    number=1000
)

print(f"Untyped: {time_untyped:.4f}s")
print(f"Typed: {time_typed:.4f}s")
# Performance difference is negligible

    Conclusion

    Type hints transform Python from a "hope it works" language into a "know it works" language. They serve as executable documentation, catch bugs before they reach production, and make code self-explanatory to both humans and tools.

    The clean-py repository demonstrates that comprehensive typing doesn't sacrifice Python's expressiveness—it enhances it. By making implicit assumptions explicit, type hints create more reliable, maintainable, and professional Python applications.

    Key benefits of comprehensive type hints:

    • Early Bug Detection: Catch errors during development, not production
    • Better IDE Support: Auto-completion, refactoring, and navigation
    • Self-Documenting Code: Types explain intent better than comments
    • Refactoring Safety: Large-scale changes become less risky
    • Team Communication: Clear contracts between code components

    Start small with your most critical domain models and gradually expand coverage. Modern Python's type system is powerful enough to express complex domain logic while remaining readable and maintainable.
